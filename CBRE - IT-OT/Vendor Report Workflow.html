CBRE IT/OT — Vendor Report
Maps OT applications and vendors across properties — showing which systems are installed at each building
Trigger

Armis Data Sync

Scheduled Armis API pull via n8n. Data lands in S3 at cbre/armis-data/

On-Demand Request

Bhruti or CBRE stakeholder requests updated vendor mapping for a property or portfolio view.

Inputs

Armis API Data

Primary source — device and application inventory from Armis scans

Applications (name, version, manufacturer)
Devices (type, boundary, site)
Boundaries (network segments)
Source: brookfieldproperties.armis.com

Vendor-Boundary Mapping

Curated reference data linking vendors to OT categories and boundaries

Fields: site short, OT building category, vendor, boundary ID, boundary name

Source: Airtable export (manual curation)
Linear: CUS-706
Transformation

Ingest & Extract

Pull Armis JSON from S3. Extract application names, device info, and property/site associations. Parse boundary information for vendor grouping.

OT Classification & Filtering

Filter out non-OT software from raw Armis application lists. Classify remaining applications into OT building categories: Access Control, BMS, Surveillance, Elevators, Lighting, Metering, etc. LLM-assisted classification for ambiguous entries.

Vendor Mapping & Aggregation

Join Armis data with vendor-boundary mapping. Use boundary as the join key to link devices to vendors and OT categories. Handle multi-boundary vendors. Aggregate into hierarchical view: Region → Property → OT Category → Application → Vendor.

Outputs

Vendor Intelligence Report

Excel artifact — tabular view per property

Columns: Region, Property, OT Building Category, Application/System, Vendor

Shows which systems are installed at each building, which vendors support them, and where regional inconsistencies exist.
